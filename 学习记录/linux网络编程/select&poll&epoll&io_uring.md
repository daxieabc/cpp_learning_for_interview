select  
select 是一个核心的系统调用（system call），主要用于 I/O 多路复用（I/O Multiplexing）  
select 让程序能够阻塞（暂停执行），直到以下情况之一发生：  
被监视的文件描述符集合中，有一个或多个变得可读（例如：socket 接收到数据，或者管道有数据）。  
被监视的文件描述符集合中，有一个或多个变得可写（例如：socket 发送缓冲区有空闲空间）。  
被监视的文件描述符集合中，有一个或多个发生异常。  
超过了指定的超时时间。  
使用方法：  
1、创建两个位图，一个记录了我需要监听的FD（file description，socket也是一种fd）  
另一个传给select，select会监视所有位图标记的FD，并根据是否有数据写入，修改位图  
fd_set master_fds;  // 【主副本】：永远保存你想要监听的所有 FD  
fd_set read_fds;    // 【工作副本】：每次传给 select 用  
2、遍历一下返回的位图，如果监听的位图设置为1，就通过socket接受连接
  


poll  
和select是同一原理，只不过将位图换成了结构体，结构体如下：  
struct pollfd {  
    int   fd;      // 【输入】文件描述符 (File Descriptor)  
    short events;  // 【输入】你感兴趣的事件 (例如：可读、可写)  
    short revents; // 【输出】实际发生的事件 (内核返回的结果)  
};  
比较select的好处：  
打破了 1024 的限制：可以监听更多连接（受限于系统内存和最大文件句柄数）。  
接口更友好：使用数组和结构体，避免了繁琐的位操作 (FD_SET, FD_ISSET) 和每次调用前的重置麻烦。  
事件更丰富：能区分更多种类的系统事件（如断开连接、优先级数据等）  


  
epoll  
里面有两个东西，一个是红黑树，用来存储要监听的FD和事件等，另一个是结构体链表，用来返回给调用者处理。有两种触发模式：水平触发（只要 FD 缓冲区里有数据，epoll_wait 就一直通知），边缘触发（FD 状态发生变化才通知）  
使用流程：  
第一步：创建实例  
第二步：注册/修改/删除 FD (核心控制)  
第三步：等待事件  

  
io_uring
零系统调用（批处理）：通过共享内存环形队列提交任务和获取结果，无需每次陷入内核。  
真正的异步（Proactor 模式）：你只管提交“读/写”命令，内核做完后通知你，无需关心“是否就绪”。  
高级特性：支持链接操作、超时、固定文件描述符、直接 I/O 等。
